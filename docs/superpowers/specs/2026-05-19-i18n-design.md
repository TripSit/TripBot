# i18n System Design Spec

**Date:** 2026-05-19
**Branch:** i18n-wip
**Status:** Approved

---

## 1. Scope

### In scope

Every user-visible string in every Discord command layer ŌĆö slash command names, option descriptions, embed titles, embed field names and values, embed footers, reply content, modal labels ŌĆö must be wrapped with `t()` and registered with `getCommandLocalizations()`.

Affected file types: `d.*.ts`, `m.*.ts`, `u.*.ts` under `src/discord/commands/`.

### Out of scope

The following strings are **never** localized:

- Log messages (`log.info`, `log.error`, etc.)
- Sentry / Rollbar error strings
- Developer-facing error messages
- All strings in admin and mod commands: `d.admin.ts`, `d.mod.ts`, `d.botmod.ts`, `d.bottest.ts`, `d.botstats.ts`, `d.clearchat.ts`, `d.purge.ts`

No `t()`, `getLocale()`, or `getCommandLocalizations()` calls are added to admin, mod, or developer-facing code.

### Deferred (spec-defined, implementation deferred)

The following will be localized in a future cycle using the same rules defined in this spec:

- `d.rpg.ts` (4,157 lines) ŌĆö deferred due to size
- The 7 admin/mod commands listed above ŌĆö deferred as out of scope for user-facing i18n

## 2. Namespace and Key Rules

### File layout

```
src/locales/
  en-US/          ŌåÉ source of truth
    common.json
    <command>.json
  fi/
    common.json
    <command>.json
  <locale>/
    ...
```

One JSON file per command namespace. `en-US` values are authoritative; all other locales are derived from it.

### `common.json`

Holds strings that are either:
- Shared across 2 or more commands, or
- Reserved/generic keywords (`yes`, `no`, `errorUnexpected`, and similar)

Command namespace files must not duplicate a key already defined in `common.json`.

### Key conventions

| What | Convention | Example |
|------|------------|---------|
| General keys | camelCase | `karmaMsg` |
| Required keys | every namespace must include | `commandName`, `commandDescription` |
| Option description keys | `<optionName>Option` | `targetOption`, `ephemeralOption` |
| Interpolation | `{{varName}}` in values | `"{{name}} has {{count}} karma"` |

### Sync tool

`npm run i18n:sync` is the enforcer:
- Adds missing keys to non-`en-US` locales (empty string placeholder)
- Removes stale keys no longer in `en-US`
- Reorders keys to match `en-US` order
- Never overwrites a non-empty translation

`npm run i18n:sync -- --dry` previews changes without writing. This is the verification step after every command is localized.

### Definition of "fully localized"

A command is fully localized when all three conditions hold:

1. Every user-visible string is wrapped in `t(locale, namespace, key, vars?)`
2. Every `.setName()` and `.setDescription()` call on the command and its options has a matching `getCommandLocalizations(namespace, key)` call
3. `npm run i18n:sync -- --dry` reports no drift

## 3. Runtime Locale Resolution

Locale is resolved per-guild at interaction time by `getLocale(interaction, namespace)`.

**Priority order:**
1. `discord_guilds.locale` from the database (guild admin setting)
2. `env.LOCALE` environment variable
3. `en-US` (hard default)

A **valid locale** is any directory name under `src/locales/` matching `/^[a-zA-Z-]+$/`.

No runtime validation of the locale value is performed beyond the DB lookup. If a locale directory or key is missing, i18next falls back to `en-US` automatically via `fallbackLng`.

---

## 4. `getCommandLocalizations` and Discord Registration

- All slash command `.setName()` and `.setDescription()` calls, and every option's `.setDescription()`, must be paired with the corresponding `getCommandLocalizations(ns, key)` call.
- `getCommandLocalizations` reads all non-`en-US` locale directories at command-registration time. If a key has an empty string value in a locale file (untranslated), that locale is omitted from the map ŌĆö Discord falls back to `en-US`.
- `npm run tripbot:deployCommands` must be run after any command is localized, since option descriptions are registered with Discord at deploy time, not at runtime.

## 5. `setup locale` Subcommand

A new `locale` subcommand is added to `src/discord/commands/guild/d.setup.ts`.

### Access control

Restricted to guild admins, consistent with other `setup` subcommands.

### Subcommands

| Subcommand | Behavior |
|------------|----------|
| `/setup locale get` | Replies with the guild's current locale, or `en-US` if none is set |
| `/setup locale set <locale>` | Validates the value against `src/locales/` directories, writes to `discord_guilds.locale`, replies with confirmation. Invalid values get an error reply listing available locales. |

### Autocomplete

The `locale` string option on `set` uses autocomplete to enumerate valid locale directories at interaction time ŌĆö no hardcoded list.

### Localization

`src/locales/en-US/setup.json` is extended with all new strings for this subcommand. All new strings follow the same rules as every other command.

## 6. Contributor Workflow for New Locales

To add a new locale:

1. Create `src/locales/<locale>/` directory
2. Run `npm run i18n:sync` to generate all namespace files with empty values
3. Translate all strings from `en-US` using any AI tool. Claude Opus 4.7 with up to 20 parallel agents is suggested for quality and performance, but any tool may be used.
4. Submit a PR to `tripsit/TripBot` upstream

### Finnish (`fi`) locale

New namespace files for `fi` are filled by a Psyvault member with repository access using the same workflow, then pushed to the Psyvault fork and upstream.

### Translation quality review

Translation quality is reviewed by TripSit members with repository access who speak the language, or by using a TripSit community-provided tool to generate a translation quality and accuracy report.
